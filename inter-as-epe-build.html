Why a square not a triangle: a triangle gives every node a trivial loop-free alternate, so it under-demonstrates TI-LFA. The square has no strict PQ-node → the repair list grows to a 2-SID stack (the real FRR case). It also gives two equal-cost paths for ECMP/UCMP and a long-vs-short choice for SR-TE/flex-algo.

STAGE 1 — AS 100 SR core, as a SQUARE (PE1, P1, P3, ASBR1)

Proves on real IOS-XR: §3 baseline SR (prefix-SIDs, label-FIB), §4 TI-LFA (square → 2-SID repair), §5 flex-algo, §6 SR-TE explicit path, §8 ECMP/UCMP. The square (two disjoint PE1→ASBR1 paths) is what makes the steering/FRR demos possible — a line couldn't.

Strategy in one line: build the square so PE1 has a short path (via P1) and a long path (via P3) to ASBR1. Baseline SR ECMPs both; then we set metrics so one is "short", and use flex-algo / SR-TE to deliberately ride the OTHER — that's the capture. TI-LFA: pull the active link, the repair list steers onto the disjoint path.

Build in EVE

  1. Open Test.unl (or a new lab). Add 4× xrv9k nodes — PE1, P1, P3, ASBR1.
  2. Wire the square (PE1 and ASBR1 NOT directly connected):
    • PE1 Gi0/0/0/0 ── P1 Gi0/0/0/0
    • P1 Gi0/0/0/1 ── ASBR1 Gi0/0/0/0
    • PE1 Gi0/0/0/1 ── P3 Gi0/0/0/0
    • P3 Gi0/0/0/1 ── ASBR1 Gi0/0/0/2
  3. Start the 4 nodes. Wait ~4 min for each ...CPU0:ios# prompt.

STAGE 2 — AS 200 + Inter-AS Option C boundary

Proves: deck §11 core — two SR domains stitched by BGP-LU (transport label across the boundary) + multihop eBGP VPNv4 (the L3VPN service), next-hop-unchanged so the egress PE loopback survives.

Build in EVE

  1. Add 3× xrv9k: ASBR2, P2, PE2.
  2. Wire AS 200 internally: ASBR2 Gi0/0/0/1 ── P2 Gi0/0/0/0; P2 Gi0/0/0/1 ── PE2 Gi0/0/0/0.
  3. Inter-AS link: ASBR1 Gi0/0/0/1 ── ASBR2 Gi0/0/0/0.
  4. Configure AS 200 internal IS-IS+SR exactly like AS 100 (net 49.0002..., SIDs 4/5/6) — same template, change loopbacks/SIDs/IPs.

Reachability to the far loopback: for the multihop VPNv4 session (PE1 ↔ PE2 loopbacks) to come up, each PE must have a route to the other's loopback. STAGE 3 — Egress Peer Engineering (EPE)

Proves: deck §11 EPE — ASBR1 allocates a PeerNode-SID for its eBGP peer PEERX (AS 300), exported via BGP-LS; head-end can steer egress out the chosen peer. The one place SR is non-debatable inter-AS.

Build in EVE

  1. Add 1× xrv9k: PEERX. Wire ASBR1 Gi0/0/0/3 ── PEERX Gi0/0/0/0. (Gi0/0/0/3 — because Gi2 now goes to P3.)
  2. Link: ASBR1 Gi0/0/0/3 = 10.1.37.3/30, PEERX Gi0/0/0/0 = 10.1.37.7/30. PEERX Lo0 = 10.0.0.7, AS 300.
