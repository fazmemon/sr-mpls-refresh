Inter-AS Option C + EPE — EVE-NG build

Platform: CloudMyLab EVE-NG Pro trial, image xrv9k-fullk9-7.4.2 (IOS-XR 7.4.2, full SR). Pod = 48 vCPU / 251.8 GB RAM / KSM on → 8× XRv9000 (16 GB each ≈ 128 GB) fits comfortably.
Goal: real captures for deck §11 (Inter-AS Option C + EPE), and en route §1–§4 SR baseline / TI-LFA / flex-algo on XR.
Console: telnet (HTML console in the web UI). XR boots slowly (~3–5 min); wait for the RP/0/RP0/CPU0:ios prompt and run commit after each config block.

XR config workflow reminder: IOS-XR is two-stage — type config, then commit. Nothing applies until committed. show configuration failed if a commit is rejected. Each node's loopback must be enrolled in IS-IS (passive under the interface) or its prefix-SID won't advertise — the exact trap hit on the cEOS lab.

STAGE 1 — AS 100 SR core (PE1 ─ P1 ─ ASBR1)

Proves: IS-IS + SR-MPLS baseline on XR, prefix-SIDs, label-FIB, TI-LFA. This alone re-captures deck §3/§4 on real IOS-XR (vs the cEOS captures).

Build in EVE

  1. Open Test.unl (or a new lab). Add 3× xrv9k nodes — name them PE1, P1, ASBR1.
  2. Wire: PE1 Gi0/0/0/0 ── P1 Gi0/0/0/0; P1 Gi0/0/0/1 ── ASBR1 Gi0/0/0/0.
  3. Start the 3 nodes. Wait ~4 min for each ...CPU0:ios# prompt.

STAGE 2 — AS 200 + Inter-AS Option C boundary

Proves: deck §11 core — two SR domains stitched by BGP-LU (transport label across the boundary) + multihop eBGP VPNv4 (the L3VPN service), next-hop-unchanged so the egress PE loopback survives.

Build in EVE

  1. Add 3× xrv9k: ASBR2, P2, PE2.
  2. Wire AS 200 internally: ASBR2 Gi0/0/0/1 ── P2 Gi0/0/0/0; P2 Gi0/0/0/1 ── PE2 Gi0/0/0/0.
  3. Inter-AS link: ASBR1 Gi0/0/0/1 ── ASBR2 Gi0/0/0/0.
  4. Configure AS 200 internal IS-IS+SR exactly like AS 100 (net 49.0002..., SIDs 4/5/6) — same template, change loopbacks/SIDs/IPs.

Reachability to the far loopback: for the multihop VPNv4 session (PE1 ↔ PE2 loopbacks) to come up, each PE must have a route to the other's loopback. BGP-LU between ASBRs provides it: ASBR redistributes/advertises the far loopbacks as labeled-unicast. If the VPNv4 session won't establish, that's the missing transport route — check show route 10.0.0.6 on PE1.

STAGE 3 — Egress Peer Engineering (EPE)

Proves: deck §11 EPE — ASBR1 allocates a PeerNode-SID for its eBGP peer PEERX (AS 300), exported via BGP-LS; head-end can steer egress out the chosen peer. The one place SR is non-debatable inter-AS.

Build in EVE

  1. Add 1× xrv9k: PEERX. Wire ASBR1 Gi0/0/0/2 ── PEERX Gi0/0/0/0.
  2. Link: ASBR1 Gi0/0/0/2 = 10.1.37.3/30, PEERX Gi0/0/0/0 = 10.1.37.7/30. PEERX Lo0 = 10.0.0.7, AS 300.

ASBR1 — enable EPE on the peer + BGP-LS

configure
router bgp 100
 neighbor 10.1.37.7
  remote-as 300
  egress-engineering                 ! allocate PeerNode-SID for this peer
  address-family ipv4 unicast
   route-policy PASS in
   route-policy PASS out
  !
 !
 ! export topology + EPE SIDs to a controller via BGP-LS
 address-family link-state link-state
 !
!
! distribute IS-IS + EPE into BGP-LS
router isis CORE
 distribute link-state
!
commit
end
