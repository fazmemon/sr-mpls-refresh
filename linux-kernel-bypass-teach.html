Reading the counters = the diagnosis

Which counter names which stage

  • rx_dropped rising, CPUs idle — ring too small for the burst. Grow it: ethtool -G.
  • ring drops with one core pegged in softirq — the CPU can't drain fast enough. Spread with RSS/RPS, pin the IRQ off the app core, raise netdev_budget. time_squeeze (softnet col3) is the tell.
  • RcvbufErrors rising, CPU idle — socket buffer overran: the app reads too slowly. Raise rmem_max + SO_RCVBUF, then fix the reader.
  • InErrors ≥ RcvbufErrors always — a buffer-full drop charges both. InErrors rising without RcvbufErrors = corrupt / no-listening-port, NOT gapping.
The triage tree, one line: a gap on ONE consumer = host problem · gaps on ALL consumers of one line = that line/network · both lines everywhere = upstream of you. Decide that BEFORE touching a CLI — it picks the layer for free.

The stage that matters — gapping lives here

The socket buffer — the experiment to run cold

Market-data gapping is almost always this stage: bursts arrive fine, the app drains too slowly, UDP drops locally, sequence numbers go missing. Reproduce it end-to-end on the EC2 box yourself — these are the target numbers, not ones you've produced yet:

bufferapp receivedRcvbufErrors
tiny buffer + slow reader16 KB1,248 / 300,000+298,752
32 MB buffer + reader keeping up64 MB300,000 / 300,0000
  • OutDatagrams climbed by exactly 300,000 — the packets reached the host. The loss was local.
  • The silent villain: net.core.rmem_max defaults to ~208 KB and silently caps whatever SO_RCVBUF the app asks for — ask for 32 MB, get 425 KB, never told.
  • Fix order: raise the ceiling (sysctl net.core.rmem_max), raise the ask (SO_RCVBUF), size to burst × rate.
Why a big buffer is NOT the real fix: it absorbs a burst, not a sustained rate mismatch. If arrival > drain, any finite buffer fills — the buffer is a shock absorber. The real fix is a faster consumer: pin it off the IRQ core, busy-poll, or bypass the kernel entirely.
4 · Socket stage4
The code behind the drill — write it cold, deck is the check

The sender, the receiver, and the counter

Receiver — small buffer, reads slowly, so it overruns:

import socket, time
s = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
s.setsockopt(socket.SOL_SOCKET, socket.SO_RCVBUF, 16384)
s.bind(("127.0.0.1", 9999))
print("SO_RCVBUF =", s.getsockopt(socket.SOL_SOCKET, socket.SO_RCVBUF))
n = 0
while True:
    s.recvfrom(2048); n += 1
    if n % 500 == 0: time.sleep(0.05)   # slow reader = overrun

Sender — flood, no throttle (a throttle would never overrun it):

import socket
s = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
for _ in range(300000):
    s.sendto(b'a' * 1400, ("127.0.0.1", 9999))
print("sent")

Watch it drop, then fix it:

nstat -n                    # snapshot baseline (silent)
#   ...run the sender...
nstat -a | grep -i rcvbuf   # UdpRcvbufErrors = dropped at the socket
sudo sysctl -w net.core.rmem_max=33554432
#   raise ceiling (default ~208 KB caps SO_RCVBUF),
#   then set SO_RCVBUF=33554432 in receiver -> ~0 for a burst
The four verbs to own cold: socket() make it · bind() claim a port (receiver only) · recvfrom() / sendto() in a loop. Everything else — buffer size, payload, port — you invent each time. Constants live in the module: socket.AF_INET, socket.SOCK_DGRAM. And nstat -n snapshots silently; plain nstat / -a shows the count.

The low-latency host — before bypass

The tuning checklist (know why each exists)

Isolate the hot cores

isolcpus + nohz_full + rcu_nocbs — dedicate cores, kill the scheduler tick. SMT off (a sibling on the shared physical core = contention = jitter). Governor = performance, deep C-states off.

Keep interrupts away

Pin NIC-queue IRQs to housekeeping cores (smp_affinity), never the app core. Coalescing DOWN for latency (rx-usecs), ring UP for burst, GRO/LRO off when latency beats throughput.

Steer flows to the app

RSS = hardware hash spreads queues across cores; RFS = deliver a flow to the core its app runs on (cache locality). Busy-poll (SO_BUSY_POLL) — spin instead of sleeping: an idle wake ≈ 8µs, busy-poll ≈ 4µs.

Memory discipline

NIC + app on the same NUMA node. Hugepages (fewer TLB misses), mlockall, swap off. The frame: determinism — the average was never the problem, the tail is.

When the kernel itself is the tax

Kernel bypass — why, and the three ways

The normal path (IRQ → kernel stack → socket → syscall → copy) costs a few µs and jitters — every interrupt, context switch and copy is variable. For trading, the jitter matters more than the average. Bypass deletes those steps:

  • DPDK — a userspace poll-mode driver owns the NIC and spins a dedicated core. Lowest, most deterministic. You rewrite the app to its API, and that core is 100% busy forever.
  • Solarflare / Onload — transparent: shims the standard socket calls, existing app gets userspace networking with no code change. The pragmatic trading-desk default.
  • AF_XDP — the modern in-kernel middle ground: an eBPF hook at the driver redirects chosen packets to a userspace ring. Faster than the stack, keeps more kernel tooling than DPDK.
The trade to state unprompted: bypass burns whole cores (poll-mode = 100% CPU by design) and blinds normal tooling — tcpdump, ss, netfilter no longer see the traffic. You spend cores + observability to buy deterministic latency. Saying the cost is what makes the answer senior.
Choosing between them: Onload when you cannot touch the app · DPDK when you own it and the latency budget pays for a burnt core · AF_XDP when you need the speed but want to keep the kernel's tooling.

The cloud wrinkle you can speak to first-hand

EC2 / ENA — the invisible drop ceiling

ip -s link ens5   →  errors 0, dropped 0        (looks clean!)
ethtool -S ens5   →  bw_in_allowance_exceeded: 498   ← AWS shaped you at the hypervisor
  • Nitro enforces per-instance allowances — bw_in / bw_out / pps / conntrack — and when you exceed them the hypervisor drops silently: kernel counters read zero.
  • The tell: "0 drops but losing packets" on EC2 = check the ENA *_allowance_exceeded counters. ethtool -S is your only visibility.
  • Honest frame: real HFT is colocated bare metal (Solarflare + bypass + hardware timestamps). EC2 can't bypass ENA and its coalescing knobs are limited — know the difference, say it plainly.
PTP belongs in this story: you cannot optimise latency you can't measure. NTP is milliseconds; hardware timestamping (NIC PHC, ptp4l + phc2sys) is nanoseconds — it's both the measurement floor and the regulatory timestamp (RTS-25 class).

Putting it together — the round itself

"The feed is gapping" — the walk you own

  • 1 · Localise first. One consumer or all? One line or both? That alone picks host / line / upstream.
  • 2 · Wire truth. The tap (hardware timestamps, pre-socket) shows every sequence present and in order — so the network delivered. It's the host.
  • 3 · Walk the ladder up. Ring counters clean → softirq time_squeeze rising → RcvbufErrors climbing with Recv-Q pegged at the buffer cap = the socket stage, named.
  • 4 · Fix in two tiers. Immediate: raise rmem_max + SO_RCVBUF, repin the consumer. Structural: faster consumer / bypass — and note what the A/B arbitration was doing meanwhile (if BOTH feeds gapped on one host, it's that host, full stop).
  • 5 · Narrate calmly. The composure while walking counters IS the test for a trading-infra seat.
Your grounding — once you've done the drills: this exact fault is planted in the pcap drill (captures/pcap-drill-01.html), waiting for you to diagnose cold — wire complete + softnet time_squeeze + RcvbufErrors + Recv-Q pegged at the 212992 cap — and the socket-stage overrun is set up to reproduce and fix on the EC2 box. Do both and you'll be describing something you've done, not read.
