Stage 0 — decide the topology (do this first, on paper)

A single topology serves all three stages, so it's worth 10 minutes up front:

```

┌────────┐ ┌────────┐

│ spine1 │ │ spine2 │ ← SR-MPLS core, OSPF-SR or IS-IS-SR

└───┬────┘ └────┬───┘

┌──────┼──────┬───────────┼──────┐

┌───┴──┐ ┌─┴────┐ │ ┌───┴──┐ │

│ leaf1│ │ leaf2│ │ │ leaf3│ │ ← VTEPs, EVPN control plane

└───┬──┘ └──┬───┘ │ └───┬──┘ │

host1 host2 │ host3 │

```

  • 3 leaves, 2 spines is the smallest topology that shows ECMP, EVPN multi-VTEP and an SR path with a

real choice of route. Fewer and the interesting behaviour disappears.

  • Loopbacks 10.0.0.x/32 (these become both the VTEP source and the SR prefix-SID targets).
  • Point-to-point /31s between leaf and spine.
  • Check container memory before building: cEOS wants ~2GB each, so 5 nodes needs headroom.

Stage 1 — the underlay + SR-MPLS

What you're proving: that a label path exists and that you can see it.

Targets, in order:

1. Bring up the IGP across all five nodes (OSPF or IS-IS — pick one and know why you picked it).

2. Advertise loopbacks; confirm full reachability leaf-to-leaf.

3. Enable Segment Routing on the IGP, advertise an SRGB, assign a prefix-SID per loopback.

4. Verify: does leaf1 have a label for leaf3's loopback, and does it match SRGB base + index?

5. Trace the label path — confirm what the spine does to the label (swap), and what the penultimate hop

does (pop, if implicit-null).

The things to actually look at, not just configure:

  • The label table. SRGB base + index should be predictable — if it isn't, your SRGB isn't uniform.
  • What happens to the label stack hop by hop. This is the difference between knowing SR and reciting it.

Failure to expect: cEOS SR-MPLS support varies by version. Verify SR is supported on your image

before building a plan around it — if it isn't, the EVPN and SDN stages still stand on their own.

Stage 2 — EVPN/VXLAN overlay

What you're proving: the control plane and data plane are separate things, and you can see both.

1. eBGP or iBGP for the EVPN address family (choose, and be able to defend the choice).

2. Configure VTEPs on the leaves, source from loopback.

3. A VLAN/VNI mapping, hosts in the same subnet across two leaves.

4. Ping host1 → host3. Then look at what made it work:

- the type-2 MAC/IP route in the EVPN table — who advertised it, what's the next-hop, what RT

- the VXLAN encapsulation on the wire (this is where the pcap work starts — see Stage 4)

5. Then break it deliberately: shut a link, watch the withdrawal and reconvergence.

Extension if there's time: symmetric IRB across two VNIs, which is the bit most people fumble.

Stage 3 — the SDN layer (your chosen items 1–3, plus 4 if time)

This is the round you're weakest on, so it's the highest-value stage.

### 3a · gNMI + OpenConfig with gnmic (item 1)

1. Verify first: does your cEOS image support the gNMI agent? (management api gnmi, and check the

transport/port). If not, this stage moves to a SONiC or SR Linux node instead — don't force it.

2. Install gnmic (single Go binary, no dependencies).

3. gnmic capabilities against a leaf — see which YANG models the box actually supports. This alone

answers "OpenConfig vs native models" from having looked, not from reading.

4. gnmic get an OpenConfig path — start with interface state, e.g.

/interfaces/interface[name=Ethernet1]/state.

5. gnmic set — change something small (a description) through the model rather than the CLI, then

confirm on the CLI that it landed. That round trip is the whole point of model-driven config.

### 3b · Streaming telemetry (item 2)

1. gnmic subscribe on interface counters, sample-interval sub-second.

2. Generate traffic across the fabric while it streams.

3. Watch the counters move in real time, then contrast: poll the same counter by CLI every 30s and

see what the average hides. That's slide 7 of the SDN deck, observed rather than asserted.

### 3c · NetBox as source of truth + Ansible (item 3)

1. NetBox via Docker (netbox-docker), free.

2. Model the lab in it — devices, interfaces, IPs, VLANs/VNIs.

3. An Ansible playbook that renders config from NetBox and pushes via eAPI or gNMI.

4. Then the part that matters: change something on a switch by hand, re-run in check mode, and watch

the drift appear as a diff. That's the reconciliation loop from the SDN deck, made real.

### 3d · SONiC node (item 4, optional)

Add a SONiC container to the topology. Look at config_db.json and the redis-backed config model —

it's a genuinely different way to hold configuration, and almost nobody interviewing has touched it.

Stage 4 — pcap on your own fabric

This closes the thinnest gap after SDN, and it costs almost nothing once the lab is up.

1. tcpdump on a leaf uplink while host1 pings host3 — decode the VXLAN encapsulation: outer MAC/IP,

UDP 4789, VNI, then the original frame.

2. Check the outer UDP source port across several flows — it's a hash of the inner flow and it is the

only entropy ECMP has. If it's constant, everything pins to one path.

3. Capture the EVPN BGP updates during a link failure — watch the withdrawal in the control plane.

4. If SR came up in Stage 1, capture the label stack and read it against the format in drill-bank W1.

5. Then do captures/pcap-drill-01.html cold — the planted market-data gapping fault. It's built and

waiting; diagnose it network-vs-host without help.
