The vocabulary (it's naming, not technology)

"Elastic" is branding — it means software-defined, detachable, scale-on-demand. It is not a technology, and the Elastic-somethings are unrelated to each other beyond the word.

The bit that isn't networking — it's geography

Region · Availability Zone · subnet

AWS= your worldThe one thing
Regiona metro you built in — London, Dublinseparate estate, separate address plan; joined by a WAN link with real latency (ms)
Availability Zonea separate DC building in that metroown power + cooling, km apart, dark fibre between, sub-millisecond
Subneta segment inside ONE buildingcannot span buildings — pinned to one AZ, by design
ENIthe NIC — a virtual interface in a subnetholds the private IP, MAC and security groups; instances, NAT GWs, endpoints and TGW attachments all get one
As geography: region = a metro · AZ = a building in that metro · subnet = a segment in one building. A subnet cannot span buildings — AWS forcing the design question of losing one. Same shape as Sky into Ireland: two sites per metro, a subsea crossing between metros.

Connecting VPCs + on-prem

Peering · Transit Gateway · Direct Connect

  • VPC peering = a point-to-point link between two VPCs. Doesn't scale — N VPCs = an N² mesh.
  • Transit Gateway (TGW) = a cloud hub / route-reflector — hub-and-spoke for many VPCs + on-prem. The fix for the peering mesh. You've designed exactly this topology in the physical world.
  • Direct Connect (DX) = a private circuit into AWS — a leased line / cross-connect, BGP over it. DX + TGW = the hybrid edge. You've turned up hundreds of BGP peerings — say so.
  • Site-to-Site VPN = IPsec over the internet — the cheaper/backup path to DX (local-pref out, more-specific back, withdraw-on-failure — your DX-primary/VPN-backup answer).
The framing that wins: none of this is new to you — it's DCI and hybrid WAN with AWS names. TGW = hub-and-spoke, DX = a BGP circuit, VPN = IPsec backup.

The design question — metro to metro

Region to region: a TGW each, peered

  • A TGW is regional — it cannot span regions. So it's one TGW per region, plus a peering attachment between them, running over AWS's own backbone (private, encrypted, no internet path).
  • Attach the VPC in EVERY AZ you use. The attachment plants an ENI in one subnet per AZ you select — an instance in building A leaves via building A's ENI and stays local. Attach one AZ only and everything hairpins through it: cross-AZ charges, and that building becomes a failure domain.
  • Routing across the peering is STATIC — route propagation doesn't run over a peering attachment, so you write the remote region's CIDR into each TGW route table yourself. Worth re-checking against current docs before quoting it, but that's been the behaviour, and it surprises everyone who lives in BGP.
  • Non-transitive across regions — A↔B and B↔C peered does NOT give you A↔C. Mesh the TGWs, or move to Cloud WAN, the managed wrapper for exactly this.
The simpler alternative, for completeness: inter-region VPC peering, direct VPC to VPC, no hub — fine for one or two, but it's the N² mesh again the moment there are several (slide 5). And cost is the design driver: cross-region transfer + per-attachment-hour + per-GB processed on the TGW.

Where you out-depth them (Nscale-relevant)

Cluster placement groups + EFA — the GPU fabric angle

  • Cluster placement group = AWS packs instances physically close (same rack/spine) for low, consistent latency — the "rail-optimised" idea, AWS-side.
  • EFA (Elastic Fabric Adapter) = a kernel-bypass NIC (RDMA-style, OS-bypass) so GPU collective ops (all-reduce) skip the network stack — AWS's answer to InfiniBand/Spectrum-X.
  • This is the same physics as the AI-fabric backend and as HFT kernel bypass.

How to handle it in the room

When AWS specifics get deep — pull up a layer

  • Lead with the primitives you own (VPC/subnet/route-table/IGW = routing-domain/segment/RIB/edge) — that's authority, not memorisation.
  • When it gets deep into AWS-native services you don't have cold, pull it up to design / topology / why — that's your ground, not theirs. "At the primitive level it's the same routing problem; the AWS service is just the packaging."
  • Don't fake depth on AWS-native managed services — be curious, ask about their estate.
Questions for them: how their VPC and Transit Gateway estate is laid out, and whether they run EFA for the GPU fabric.
Both interviews: Nscale's director is AWS-certified (be fluent + the EFA angle). HRT — if AWS comes up it's the hybrid/edge, same map. One study, both rooms.
