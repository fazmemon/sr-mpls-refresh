SR-MPLS forwarding — sequence

One label pushed at PE1, popped at penultimate hop P (PHP). No per-LSP state in P — just one entry per remote prefix-SID.

IS-IS flooding — how SR info reaches every node

graph LR PE1((PE1
node-SID 1)) P((P

node-SID 3)) X((X
node-SID 4)) PE1 ---|"LSP(PE1): SR Cap, prefix-SID, adj-SIDs"| P P ---|"reflood"| PE2 P ---|"reflood"| X PE2 -.->|"its own LSP: SR Cap + prefix-SID"| P X -.->|"its own LSP"| P

Each node originates an LSP carrying its SR Capability TLV (242), prefix-SID sub-TLV (in 135/236), and adj-SID sub-TLV (in 22). LSPs flood hop-by-hop. Every node ends with a full LSDB containing every other node's SR info — no LDP sessions, no separate label-binding protocol.

OSPF flooding — opaque LSAs carry SR info

graph LR R1((R1)) ---|"Type-10 Opaque LSA (Router Info + Ext Prefix + Ext Link)"| R2((R2)) R2 ---|"flood within area"| R3((R3)) R2 ---|"flood within area"| R4((R4)) R3 -.->|"its own Type-10 LSAs"| R2 R4 -.->|"its own Type-10 LSAs"| R2

OSPF uses opaque LSAs (RFC 5250) to carry the SR sub-TLVs — no new LSA types added. Type-10 LSAs are area-scoped (most common); type-11 LSAs are AS-scoped. Same flooding mechanism as IS-IS, just different LSA encoding. Pick IS-IS if you can — SR-related TLVs are more mature and multi-topology is cleaner.

BGP-LS → PCE — single domain

graph LR subgraph IGP[One IGP domain · one AS] R1((R1)) --- R2((R2)) --- R3((R3)) end R2 ==>|"iBGP BGP-LS (AFI 16388 / SAFI 71)"| PCE[PCE / controller]

Any one router (here R2 = the BGP-LS speaker) runs the IGP and exports the whole link-state DB — nodes, links, prefixes + SR info — to the PCE as BGP-LS NLRI. Single AS ⇒ iBGP. This is read-only topology to the controller — not labels (that's BGP-LU, later).

BGP-LS → PCE — multi-domain (H-PCE)

graph LR A[Domain A · AS 1] ==>|"iBGP BGP-LS"| cA[Child PCE A] B[Domain B · AS 2] ==>|"iBGP BGP-LS"| cB[Child PCE B] cA -->|"H-PCE (RFC 6805)"| P[Parent PCE] cB -->|"H-PCE"| P P -->|"inter-domain SR Policy · BSID at ASBR"| H[Head-end PE]

One BGP-LS speaker per domain feeds its child PCE (iBGP within each AS; eBGP-LS only if the controller sits in another AS). H-PCE stitches the cross-domain path; the head-end PE gets one SR Policy whose binding-SID hides the next domain's segment list.

Seamless MPLS — before SR

graph LR subgraph AccA[Access IGP] A1((PE-A1)) A2((PE-A2)) end subgraph Agg[Aggregation IGP] G1((AGG1)) G2((AGG2)) end subgraph Core[Core IGP] C1((C1)) C2((C2)) end subgraph AccB[Access IGP] B1((PE-B1)) end A1 --- G1 A2 --- G1 G1 ==>|"BGP-LU stitch"| C1 C1 --- C2 C2 ==>|"BGP-LU stitch"| G2 G2 --- B1

Three separate IGP domains (Access / Aggregation / Core). BGP-LU stitches transport labels between them at every boundary. Vodafone / BT / NTT mobile-backhaul shape. End-to-end LSP exists but is composed of locally-allocated labels stitched at each domain edge.

Same physical fibre. Single IS-IS domain (e.g. L1+L2 with route-leaking). Prefix-SIDs flood globally. Head-end pushes one label (Node-SID of egress) — `label = SRGB.start + 3 = 16003`. No stitching. No BGP-LU at internal boundaries. Mapping-server bridges any remaining LDP-only nodes during migration.

BGP-LU + SR — end-to-end label flow (3/3)

sequenceDiagram participant PEa as PE-A (AS 100) participant ASBRa as ASBR-A participant ASBRb as ASBR-B participant PEb as PE-B (AS 200, node-SID 5) Note over PEb: Originate BGP-LU for loopback
Prefix-SID attr: Label-Index = 5 PEb->>ASBRb: BGP-LU (label X, Prefix-SID idx=5) ASBRb->>ASBRa: eBGP-LU, NHS, Prefix-SID idx=5 preserved ASBRa->>PEa: BGP-LU (label Y, Prefix-SID idx=5) Note over PEa: SRGB.start = 16000

The BGP-LU labels (X, Y) at each hop don't have to be consistent — what matters is the Prefix-SID *index* (5) is preserved end-to-end. Each receiver computes its own label from its own SRGB.

SR-LDP interworking — label swap at the boundary

graph LR PE1["PE1 (SR)"] BN["Boundary node
SR + LDP · LFIB swaps"] PE2["PE2 (LDP-only)"] PE1 -->|"push 16005 (SID 5)"| BN BN -->|"swap 16005 → LDP 100177"| PE2 PE2 -->|"push LDP label → PE1"| BN BN -->|"swap LDP → 16001 (SID 1)"| PE1

Mapping server gives the control plane a SID for the LDP-only nodes; the boundary node (SR + LDP) does the dataplane swap both ways in its LFIB — SR label ↔ LDP label. sr-prefer makes SR win where both protocols advertise the same prefix (else LDP wins, defeating the migration). And the mapping server alone isn't enough — the LDP-only node can't pop a label it doesn't know → mapping server + boundary swap = both halves. Verify on the boundary: show mpls forwarding labels 16001 detail.

Gen 1 — RSVP-TE FRR: the pre-built bypass

graph LR HE(("Head-end")) ==> PLR(("PLR")) PLR -. "protected link FAILS" .-> N(("next-hop")) N ==> MP(("merge point")) MP ==> TAIL(("tail")) PLR == "pre-signalled bypass LSP" ==> MP linkStyle 1 stroke:#ce9178,stroke-width:3px,stroke-dasharray:6 4 linkStyle 4 stroke:#b5cea8,stroke-width:4px

Before any failure, the PLR already holds a pre-signalled bypass LSP around the protected link to a downstream merge point. The link fails → the PLR splices traffic onto the bypass in <50 ms. One bypass can protect many primaries sharing that link (facility / 1:N backup). Cost: every transit node carries RSVP bypass state.

TI-LFA on the square — what the repair actually does

graph LR p1(("p1
PLR")) pe1(("pe1")) p2(("p2")) pe2(("pe2
destination")) p1 ---|"primary — FAILS"| pe2 p1 ---|"1"| pe1 pe1 ---|"2"| p2 p2 ---|"3"| pe2 linkStyle 0 stroke:#ce9178,stroke-width:3px,stroke-dasharray:6 4 linkStyle 1 stroke:#b5cea8,stroke-width:4px linkStyle 2 stroke:#b5cea8,stroke-width:4px linkStyle 3 stroke:#b5cea8,stroke-width:4px

p1's direct link to pe2 (its primary) fails. On a square there is no loop-free LFA — p1's only other neighbour is pe1, and sending there risks looping the packet straight back.

So TI-LFA encodes the post-convergence path as a SID list: at p1, push [Node-SID p2 = 900003, Node-SID pe2 = 900004] and send out the pe1 link. Those labels steer the packet p1 → pe1 → p2 → pe2 — the exact path IGP installs once it converges, so no micro-loop.

Two SIDs, not one, because the square has no single PQ-node that sees both sides of the break. This is the lab's repair = [p2, pe2].

VPC anatomy — public vs private

graph LR NET((Internet)) IGW[Internet Gateway] subgraph VPC[VPC 10.0.0.0/16] PUB["Public subnet — route 0/0 to IGW"] NAT[NAT Gateway] PRIV["Private subnet — route 0/0 to NAT"] end NET --- IGW IGW --- PUB PRIV -->|egress| NAT NAT --> IGW

Public subnet = default route to the IGW. Private subnet egresses via the NAT GW (which lives in the public subnet) → out the IGW; nothing reaches the private instance inbound.

Every leaf connects to every spine; spines never interconnect, leaves never interconnect. Any server→server = leaf→spine→leaf = uniform 2 hops, with massive ECMP (one path per spine). No STP — it's all L3. Add spines for bandwidth, add leaves for ports.

Underlay vs overlay — the two stacked layers

graph TB subgraph OV["OVERLAY — tenant view"] direction LR SA["Server A"] -.->|"VXLAN tunnel (EVPN-signalled)"| SB["Server B"] end subgraph UL["UNDERLAY — physical IP fabric"] direction LR LA["Leaf1/VTEP"] --- S["Spine"] --- LB["Leaf3/VTEP"] end OV -. "rides on top of" .-> UL

Underlay (bottom): plain IP, eBGP carries only VTEP loopbacks. Overlay (top): the VXLAN tunnel between two VTEPs, with EVPN telling each VTEP where every MAC lives. The tenant thinks A and B share a LAN; physically it's IP + encapsulation.

VXLAN encapsulation — packet format

Ingress VTEP wraps the original frame: outer IP (VTEP→VTEP) | UDP | VXLAN+VNI | original frame — "MAC-in-UDP". Outer src-UDP-port = inner-flow hash → gives the spines ECMP entropy. Adds ~50 bytes → underlay MTU must be jumbo (9216) or you fragment/drop.

The spine is a pure IP router — it forwards on the outer VTEP-to-VTEP IP and is oblivious to tenants/VNIs. All the intelligence is at the leaves (the VTEPs). EVPN pre-populated Leaf1 with "MAC-B is behind Leaf3's loopback", so there's no flooding.

VXLAN dataplane — encapsulation

graph LR F["Original L2 frame"] --> V["+ VXLAN header (24-bit VNI)"] V --> U["+ UDP (dst 4789, src = inner-flow hash)"] U --> I["+ outer IP (src/dst = VTEP loopbacks)"] I --> W(("underlay / ECMP"))

VTEP encaps/decaps. 24-bit VNI = 16M segments. Source UDP port = inner-flow hash → ECMP entropy across the fabric. +50B overhead → underlay needs jumbo MTU (~9000).

IRB — symmetric vs asymmetric

graph TB subgraph A[Asymmetric IRB] A1["Ingress leaf: route into dest L2VNI"] --> A2["Egress leaf: bridge to host"] end subgraph S[Symmetric IRB] B1["Ingress leaf: route into L3VNI"] --> B2["Egress leaf: route out to host"] end

IRB = routing + bridging at the VTEP, with an anycast gateway (same IP+MAC on every leaf → gateway always local, seamless mobility). Asymmetric needs every VNI on every leaf; symmetric uses one common per-VRF L3VNI → scales, the modern default.

Topology — 2 spine · 2 leaf CLOS

graph TB subgraph UL[Underlay: eBGP / OSPF] S1[Spine 1] S2[Spine 2] end L1["Leaf 1 — VTEP"] L2["Leaf 2 — VTEP"] S1 --- L1 S1 --- L2 S2 --- L1 S2 --- L2 HA["Host A · Tenant 1"] --- L1 HB["Host B · Tenant 2"] --- L1 HC["Host C · Tenant 1"] --- L2 HD["Host D · Tenant 2"] --- L2

Spines carry the eBGP/OSPF underlay (every VTEP loopback reachable). Leaves are VTEPs running BGP-EVPN overlay. Hosts split across two tenants (VRFs). VXLAN tunnels run leaf ↔ leaf.
