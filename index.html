Each node originates an LSP carrying its SR Capability TLV (242), prefix-SID sub-TLV (in 135/236), and adj-SID sub-TLV (in 22). LSPs flood hop-by-hop. Every node ends with a full LSDB containing every other node's SR info — no LDP sessions, no separate label-binding protocol.

OSPF flooding — opaque LSAs carry SR info

graph LR R1((R1)) ---|"Type-10 Opaque LSA (Router Info + Ext Prefix + Ext Link)"| R2((R2)) R2 ---|"flood within area"| R3((R3)) R2 ---|"flood within area"| R4((R4)) R3 -.->|"its own Type-10 LSAs"| R2 R4 -.->|"its own Type-10 LSAs"| R2

OSPF uses opaque LSAs (RFC 5250) to carry the SR sub-TLVs — no new LSA types added. Type-10 LSAs are area-scoped (most common); type-11 LSAs are AS-scoped. Same flooding mechanism as IS-IS, just different LSA encoding. Pick IS-IS if you can — SR-related TLVs are more mature and multi-topology is cleaner.

BGP-LS → PCE — single domain

graph LR subgraph IGP[One IGP domain · one AS] R1((R1)) --- R2((R2)) --- R3((R3)) end R2 ==>|"iBGP BGP-LS (AFI 16388 / SAFI 71)"| PCE[PCE / controller]

Any one router (here R2 = the BGP-LS speaker) runs the IGP and exports the whole link-state DB — nodes, links, prefixes + SR info — to the PCE as BGP-LS NLRI. Single AS ⇒ iBGP. This is read-only topology to the controller — not labels (that's BGP-LU, later).

BGP-LS → PCE — multi-domain (H-PCE)

graph LR A[Domain A · AS 1] ==>|"iBGP BGP-LS"| cA[Child PCE A] B[Domain B · AS 2] ==>|"iBGP BGP-LS"| cB[Child PCE B] cA -->|"H-PCE (RFC 6805)"| P[Parent PCE] cB -->|"H-PCE"| P P -->|"inter-domain SR Policy · BSID at ASBR"| H[Head-end PE]

One BGP-LS speaker per domain feeds its child PCE (iBGP within each AS; eBGP-LS only if the controller sits in another AS). H-PCE stitches the cross-domain path; the head-end PE gets one SR Policy whose binding-SID hides the next domain's segment list.

Gen 1 — RSVP-TE FRR: the pre-built bypass

graph LR HE(("Head-end")) ==> PLR(("PLR")) PLR -. "protected link FAILS" .-> N(("next-hop")) N ==> MP(("merge point")) MP ==> TAIL(("tail")) PLR == "pre-signalled bypass LSP" ==> MP linkStyle 1 stroke:#ce9178,stroke-width:3px,stroke-dasharray:6 4 linkStyle 4 stroke:#b5cea8,stroke-width:4px

Before any failure the PLR already holds a pre-signalled bypass LSP around the protected link to a downstream merge point. Link fails → PLR splices onto the bypass in <50 ms. One bypass protects many primaries sharing the link (facility / 1:N). Cost: every transit node carries RSVP bypass state.

Gen 2 — IP LFA: a loop-free neighbour

graph LR P(("P / PLR")) -. "primary link FAILS" .-> N(("N")) N ==> D(("D / dest")) P == "loop-free alternate" ==> A(("A")) A ==> D linkStyle 0 stroke:#ce9178,stroke-width:3px,stroke-dasharray:6 4 linkStyle 2 stroke:#b5cea8,stroke-width:4px linkStyle 3 stroke:#b5cea8,stroke-width:4px

Pure IP, no tunnel. Neighbour A is a loop-free alternate only if dist(A,D) < dist(A,P) + dist(P,D) — A's own shortest path to D does not come back through P. On rings and squares no neighbour satisfies it → coverage 50-80%.

Gen 3 — Remote LFA: tunnel to a PQ-node

in P and Q space")) PQ ==> D(("D / dest")) linkStyle 0 stroke:#ce9178,stroke-width:3px,stroke-dasharray:6 4 linkStyle 1 stroke:#b5cea8,stroke-width:4px linkStyle 2 stroke:#b5cea8,stroke-width:4px

When no direct neighbour qualifies, the PLR tunnels (T-LDP) to a PQ-node — one both in the PLR's P-space (reachable without the failed link) and the destination's Q-space. From there, native forwarding to D. Covers 90-99%; fails when P ∩ Q is empty.

Gen 4 — TI-LFA: the repair is a SID list

graph LR PLR(("PLR")) -. "link FAILS" .-> N(("next-hop")) PLR == "push SID list" ==> T1(("transit")) T1 ==> T2(("transit")) T2 ==> D(("D / dest")) linkStyle 0 stroke:#ce9178,stroke-width:3px,stroke-dasharray:6 4 linkStyle 1 stroke:#b5cea8,stroke-width:4px linkStyle 2 stroke:#b5cea8,stroke-width:4px linkStyle 3 stroke:#b5cea8,stroke-width:4px

The PLR pushes a SID list (1-3 labels) encoding the post-convergence path — the exact path IGP installs after it reconverges. Transit nodes do ordinary SR forwarding: no bypass tunnels, no per-LSP state. 100% coverage if any post-failure path exists, and because the repair already follows the post-convergence path there is no micro-loop.

How the repair list is computed — the decision

flowchart TB A["P-space of PLR"] --> C{P and Q overlap} B["Q-space of dest"] --> C C -->|yes| D["one Node-SID PQ"] C -->|no| E["reach P-node then Adj-SID into Q-node, two SIDs"] E --> F["else add another Node-SID hop, stack grows"]

Best case is a single PQ-node (one Node-SID). No PQ-node → reach a P-node, then an Adj-SID forces the specific link into Q-space (two SIDs — the square case). The Adj-SID lets the PLR say "out this link" without disturbing the rest of the path.

Equal-cost next-hops (IGP equal metric or BGP multipath). The router hashes each flow's 5-tuple to one path — per-flow, not per-packet, so packets stay in order. Plain IP: the router reads the inner IP header directly, no label stack involved.

Failure mode #1 — asymmetric capacity

ECMP splits flows evenly by count — it has no idea the links differ in capacity. A 400G + 100G pair gets 50/50: the 100G link saturates while the 400G sits ¾ idle. Equal-cost ≠ equal-capacity → you need to weight the split (UCMP).

R reaches the destination over two links to its upstream — a 400G and a 100G — with an eBGP session on each, so it learns the prefix twice. dmzlink-bw stamps each path with its own link's speed (400, 100) — R reads that off its own interfaces, nothing is advertised to it. BGP multipath then splits the traffic in that ratio: 80% out the 400G, 20% out the 100G. Per-flow hashing, just weighted.

Step 1 — fibre availability & diversity

Demand wants 8 links — but you build only where fibre runs, and cost is subsea ≫ terrestrial ≫ metro. The trap is diversity: every path you lean on for N-1 must be physically separate. LON's links to AMS and FRA both cross to the continent — if they share a Channel duct or landing station, one anchor-drag or backhoe cuts both at once and Step 4's AMS→LON→FRA reroute never happens. "Diverse" counts only after you've SRLG-audited the ducts and landings — not when it merely looks separate on the map.

Step 2 — heavy pairs become direct links

The 8 heavy pairs → LON as a hub (4 links to its highest-demand cities: AMS, FRA, PAR, MAD) with an outer chain AMS–FRA–MIL–PAR–MAD carrying the rest. Every node has ≥2 links, so it stays connected through any single failure — though connected isn't the same as having the capacity for the reroute (Step 3). And it's 8 links, not the 15 of a full mesh.

Step 3 — light pairs don't get a link, they transit

AMS↔MAD is only 40G — not worth its own link. It rides links you already built: AMS → LON → MAD (green). Do that for all 7 light pairs and every demand rides on 8 links, not 15 — the heavy links absorb the transit. The matrix decided which links exist.

Step 4 — the N-1 budget: a built link fails

The 220 on a link is the demand, not the load. LON–AMS actually carries 350G at rest — 220 direct + 90 (AMS–PAR) + 40 (AMS–MAD) transiting it. AMS–FRA (250G) fails → AMS reaches everything only via LON, dumping AMS–FRA's 250 + AMS–MIL's 70 onto LON–AMS → 670G; LON–FRA 335 → 620G. Both ~1.6× over a 400G link. Normal peaked at 355G, so the topology passes at rest and collapses on a single failure — which is why you fix the N-1 budget before the topology and size for own load + worst inherited reroute.

Step 5 — what N-1 actually costs

Each link must hold its own load plus the worst reroute it inherits — the grey bar (490–700G). At rest nothing tops 355G, so to survive one failure you buy ~1.9× the bandwidth you use (4800G of link to carry 2520G). Backbones absorb this with dual planes — every PoP homes to two disjoint planes, so a whole plane can fail and you still hold N-1. That part is universal — every tier-1 runs it. What differs is how the reroute splits across the planes: today it's mostly IGP/ECMP even-split + capacity headroom; SR-TE / UCMP weighted paths split it proportionally so no fat pipe sits idle — the efficiency win, but most operators haven't deployed SR yet. The matrix gave you the links; the failure budget gave you the capacity and the planes.

UCMP scenario — per-service engineering

Trigger: A→B has two paths — one short-but-low-latency, one fatter. One IGP metric can't serve both classes. Give the latency-sensitive traffic (trading, voice) a min-delay Flex-Algo path and steer it there with an SR-TE colour; bulk rides the capacity path. Non-SR equivalent: parallel RSVP-TE meshes — far more state.

Where EVPN came from — pseudowire → VPLS → EVPN

The arc: point-to-point → multipoint-but-flood-and-learn → multipoint with MACs in the control plane. EVPN won by carrying MAC/IP in BGP (like it already does for IP), adding all-active multihoming + IRB, and running over MPLS or VXLAN — which is exactly the DC fabric. (Ran the VPLS→EVPN transition first-hand.)

Ingress VTEP wraps the original frame: outer IP (VTEP→VTEP) | UDP | VXLAN+VNI | original frame — "MAC-in-UDP". Outer src-UDP-port = inner-flow hash → gives the spines ECMP entropy. Adds ~50 bytes → underlay MTU must be jumbo (9216) or you fragment/drop.

The spine is a pure IP router — it forwards on the outer VTEP-to-VTEP IP and is oblivious to tenants/VNIs. All the intelligence is at the leaves (the VTEPs). EVPN pre-populated Leaf1 with "MAC-B is behind Leaf3's loopback", so there's no flooding.

IRB — symmetric vs asymmetric

graph TB subgraph A[Asymmetric IRB] A1["Ingress leaf: route into dest L2VNI"] --> A2["Egress leaf: bridge to host"] end subgraph S[Symmetric IRB] B1["Ingress leaf: route into L3VNI"] --> B2["Egress leaf: route out to host"] end

IRB = routing + bridging at the VTEP, with an anycast gateway (same IP+MAC on every leaf → gateway always local, seamless mobility). Asymmetric needs every VNI on every leaf; symmetric uses one common per-VRF L3VNI → scales, the modern default.

Topology — 2 spine · 2 leaf CLOS

graph TB subgraph UL[Underlay: eBGP / OSPF] S1[Spine 1] S2[Spine 2] end L1["Leaf 1 — VTEP"] L2["Leaf 2 — VTEP"] S1 --- L1 S1 --- L2 S2 --- L1 S2 --- L2 HA["Host A · Tenant 1"] --- L1 HB["Host B · Tenant 2"] --- L1 HC["Host C · Tenant 1"] --- L2 HD["Host D · Tenant 2"] --- L2

Spines carry the eBGP/OSPF underlay (every VTEP loopback reachable). Leaves are VTEPs running BGP-EVPN overlay. Hosts split across two tenants (VRFs). VXLAN tunnels run leaf ↔ leaf.

Overlay — VXLAN dataplane + EVPN control plane

graph LR HA[Host A] --> L1["Leaf 1 / VTEP"] L1 ==>|"VXLAN VNI 10010 over IP underlay"| L2["Leaf 2 / VTEP"] L2 --> HC[Host C] L1 -.->|"BGP-EVPN Type-2 MAC/IP"| L2

Dataplane = VXLAN tunnel leaf→leaf over the IP underlay. Control plane = BGP-EVPN: Type-2 (MAC/IP), Type-3 (flood list), Type-5 (prefix). RD scopes routes per VRF; RT controls import/export.
