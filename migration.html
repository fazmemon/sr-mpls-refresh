Honest cut: UCMP alone does not justify SR — DMZ-LB already does it. The case is steering + TI-LFA + one control plane. They asked for steering, so SR earns its place — and in a domain you own, one SR-TE policy gives steering and weighted (UCMP) candidates together.

3 · SR-MPLS vs SRv6 — the latency budget decides

On a latency/MTU-sensitive trading net, 4-byte labels beat 16-byte SIDs per hop. SR-MPLS reuses the MPLS dataplane; SRv6's network-programming upside isn't what this network needs. Pick SR-MPLS — SRv6 stays an appendix.

4 · Turning on SR-MPLS — OSPF-SR, no forklift

OSPF carries the SIDs (RFC 7684 + 8665):

  • Router-Information LSA → SRGB + SR-Algorithm
  • Extended Prefix LSA → Prefix-SID (per loopback)
  • Extended Link LSA → Adj-SID (per link)
  • ABR re-originates the Prefix-SID across areas

No LDP, no RSVP, no mapping server — the IGP already floods it.

SID plan on the in-domain core (SRGB 900000):

NodeIndexLabel
PE1 (market-data)1900001
PE2 (trading)2900002
P1 (low-latency)3900003
P2 (high-cap)4900004

label = SRGB.base + index — same meaning on every node. SRGB must match domain-wide.

From plain IP this is greenfield-easy: turn on segment-routing mpls under OSPF, allocate the SRGB, advertise prefix-SIDs. No coexistence, no flag-day.

4 · TI-LFA on the dual-plane core — sub-50 ms

PE1's primary to PE2 is via P1. P1-plane fails → TI-LFA has a pre-computed repair = the post-convergence path encoded as a SID list [900004 (P2), 900002 (PE2)], pushed at PE1. No micro-loop, no bypass state in transit, sub-50 ms with BFD. Each plane protects the other.

5 · Inter-AS — stitch domains with binding-SIDs

R-NY pushes one policy — 3 labels:

 [ 900009 | 900801 | 900802 ]
   900009  node-SID -> NYC/CORE border
   900801  BSID = CORE min-delay path
   900802  BSID = LON path to PE-LN

each BSID expands at the boundary:

 at CORE: swap 900801 -> [900013,900019]
 at LON:  swap 900802 -> [900030] -> PE-LN

Head-end carries 3 labels however deep each domain is. Controller computes from BGP-LS, pushes over BGP SR-TE (no PCEP on EOS).

6 · UCMP on the strawman — weight the unequal planes

Plain ECMP splits 50/50 → the 100G plane saturates while the 400G sits at ¼. Fix = weighted. In a domain you own: SR-TE policy, two candidate paths, weights 80/20 (proportional to 400:100). Across a boundary you don't own: BGP DMZ-Link-Bandwidth instead.

7 · Traffic steering on the strawman — class to path

Colour the market-data service colour 100 → SR-TE policy whose SID-list forces the P1 plane: [900003 (P1), 900002 (PE2)]. Bulk takes shortest-path/P2. ECMP can't express "this class, that path" — SR-TE can. (Flex-Algo would do it distributed, but it's IS-IS-only on EOS → colour policy is the OSPF answer.)

8 · Multicast — bounded (back-pocket)

SR-MPLS is unicast source-routing — it does not carry your A/B SSM feeds. Default: keep market-data multicast in its own PIM/SSM domain, off the steering fabric. If it ever needs the SR core: Tree-SID (SR P2MP) or BIER. Raise only on a segue.
