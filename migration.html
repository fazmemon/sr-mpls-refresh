What we have now — Strawman B: the multi-domain inter-AS picture

Three administrative domains, each its own OSPFv2 area 0 and its own BGP AS (65000/65001/65002 — private-ASN space for this strawman, real deployment would use the firm's allocated public ASNs). The admin boundaries are the gold eBGP links at the ASBRs — that is where end-to-end traffic engineering breaks today. Beneath each edge domain sits an eBGP-CLOS fabric of leaf/spine switches running 100s of private ASNs (16-bit private range 64512–65534), which is what inflates the ASN count into the hundreds while the number of IGP domains stays at three. Market-data exchange feeds enter at NYC; the trading desk lives behind LON. The Section 2–3 question this frames: on plain IP/eBGP there is no end-to-end path control across these boundaries today.

Should we use SR? — frame the decision honestly

Four problems are on the table. For each, name the non-SR tool we already have, find its hard limit, and only then claim SR. SR earns its keep on three of the four — not on the one everyone reaches for first (UCMP).

Problem 1 — UCMP is already solved on plain IP: BGP DMZ Link-Bandwidth

Strawman A core: PE1 reaches a destination behind P1 (400G plane) and P2 (100G plane). We want load split 400:100 = 4:1, not 1:1 ECMP. The link-bandwidth extended community carries each egress capacity; the receiving PE programs unequal FIB weights. No SR, no controller, no new label plane.

Encoding (draft-ietf-idr-link-bandwidth, verified): 8-octet extended community, Type 0x40 (non-transitive) or 0x00 (transitive), SubType 0x04, 2-octet Global-Admin (AS), 4-octet bandwidth as an IEEE-754 32-bit float in bytes/sec, not bits/sec. So 400 Gbit/s ⇒ 400e9 / 8 = 50,000,000,000 B/s in the field; 100 Gbit/s ⇒ 12,500,000,000 B/s.

Problem 1 — how DMZ-LB programs the weighted FIB (config + math)

P1/P2 carry the link-bandwidth community toward the destination (on EOS it is derived from interface speed, or tuned with link-bandwidth adjust); the receiver PE1 needs no config to consume it, only ucmp mode 1 so the parallel paths install in the FIB by ratio, not equal-cost. This is the whole UCMP story on Arista — verified feature, no SR.

! ---- PE1 (receiver) : verified UCMP path-weighting ----
router bgp 65000
   maximum-paths 4 ecmp 4                ! allow the parallel paths to install
   ucmp mode 1                           ! weighted FIB; mode 1 = integer repetition counts
   ucmp link-bandwidth update-delay 5    ! verified knob: damp churn on bandwidth *increase* (0-600s)
!
! ---- P1 / P2 (eBGP egress toward DEST) ----
! On EOS the link-bandwidth ext-community is derived from egress interface speed and
! re-advertised; no command is required on PE1 to RECEIVE it (per Arista UCMP TOI).
! Origination/adjustment knob (illustrative form - confirm exact syntax on target EOS):
!   neighbor <peer> link-bandwidth adjust auto         ! to verify on target release
!
! ---- expected verification (lab to confirm exact columns) ----
! show ip route 10.0.9.0/24
!   prints next-hops P1, P2 with FIB weight / integer repetition counts in ratio 4:1

Weight math: the receiver normalises the two LB values to integer next-hop repetition counts. 50e9 : 12.5e9 → divide by GCD 12.5e9 → 4 : 1. EOS programs P1 with 4 FIB slots, P2 with 1, so a 5-way hash sends 4/5 = 80% of flows to the 400G plane. Add a third path at 200G (25e9 B/s) and the set becomes 50e9:12.5e9:25e9 = 4:1:2 → 7 slots total, P1 = 4/7 = 57%.

Per draft-ietf-bess-ebgp-dmz the cumulative form sums downstream link-bandwidths so an upstream router weights by total reachable capacity, not just the local egress — which is exactly what a CLOS fabric beneath each domain needs.

Problem 1 — the ceiling: DMZ-LB balances by ratio, it cannot express a path

This is the crux of the whole "should we use SR" argument. DMZ-LB answers "how much down each next-hop?" It cannot answer "market-data class must take the low-latency P1 plane and bulk class must take the P2 plane." Same prefix, two classes, two intents — DMZ-LB has no knob for it.

Why no IP tool reaches it: destination-based routing (IGP/BGP) makes the forwarding decision per-prefix at every hop. To pin a class onto an explicit plane you must encode the path at the source — that is the definition of source routing, i.e. SR-TE. DMZ-LB and SR-TE solve different problems; UCMP does not imply SR.

Problem 2 — per-class steering needs source routing (SR-TE colour policy)

SR-TE expresses the explicit path as a label (SID) stack imposed by the ingress PE. To pin the P1 plane we cannot use P1's prefix-SID — that is a node-SID and follows IGP shortest path, which could ECMP back onto P2. We use the Adjacency-SID of the PE1→P1 link, then PE2's prefix-SID. Note for OSPF: EOS Flex-Algo is IS-IS-only, so per-class steering on OSPF-SR must be done with SR-TE colour policies, not Flex-Algo.

! ---- PE1 : SR-TE policy pinning the low-latency P1 plane (colour 100) ----
! Structure verified against Arista EOS Traffic-Management docs; Adj-SID label value is
! dynamic/local to PE1 -> shown as a placeholder, read it from the device, do not invent.
router traffic-engineering

      policy endpoint 10.0.0.2 color 100        ! endpoint = PE2 loopback, colour = "low-latency"
         binding-sid 965100
         path-group preference 100

SID→label is deterministic for prefix-SIDs: default EOS SR label range / SRGB is 900000–965535, so prefix-SID index 3 ⇒ label 900003, index 2 ⇒ 900002. Adjacency-SIDs are dynamic and local — their label is allocated by PE1 and must be read from the device, never invented. The colour extended community (RFC 9012) on a BGP route resolves the next-hop to the matching (endpoint, colour) policy → automated steering. This is the capability DMZ-LB structurally lacks.

Problem 3 — sub-50ms deterministic protection: LFA's gaps vs TI-LFA

Trading wants a bounded, repeatable failover. Plain IP LFA picks any loop-free neighbour — coverage is topology-dependent and the backup path is arbitrary. TI-LFA (OSPF-SR, EOS 4.33.1F) guarantees coverage and steers the repair down the post-convergence path using a precomputed SID list — so the micro-loop-free backup is also the path you'll settle on.

Method, not result: the PLR computes its extended P-space (nodes reachable from PE1's neighbours without crossing the failed link) and the Q-space (nodes that can reach the destination without crossing it), per RFC 7490 / draft-ietf-rtgwg-segment-routing-ti-lfa. It then builds the SID list along the post-convergence path: a node-SID to the P/Q boundary, plus an Adj-SID to cross the P→Q link when P and Q are not adjacent. LFA has no such guarantee — if no single neighbour is loop-free, the prefix is simply unprotected. The repair is installed in FIB before failure, so failover is data-plane-local (target sub-50ms), independent of IGP reconvergence.

Problem 4 — one control plane: SR-MPLS removes LDP and RSVP-TE

Today MPLS transport needs LDP for labels, RSVP-TE for TE tunnels, plus the IGP — three protocols, targeted-LDP sessions, and RSVP soft-state refreshed per tunnel per hop. SR-MPLS distributes labels inside the OSPF LSAs we already flood, so the label plane collapses into the IGP. Fewer protocols = fewer failure modes on a network where determinism is the product.

Where the labels come from (deterministic): the SR sub-TLVs are defined by RFC 8665 (OSPF Extensions for SR), carried in the RFC 7684 containers: the Prefix-SID sub-TLV rides the OSPFv2 Extended Prefix Opaque LSA (opaque type 7) — index 3, V/L flags clear ⇒ label = SRGB base + index = 900000 + 3 = 900003; the Adjacency-SID sub-TLV rides the Extended Link Opaque LSA (opaque type 8) and is local/dynamic. The SR-Algorithm TLV and SID/Label Range (SRGB) TLV ride the Router Information Opaque LSA (RFC 7770, opaque type 4). No separate label protocol exists in the SR path — that is the simplification, and it is structural, not cosmetic.

Verdict — SR is justified, but be precise about why

The honest line for the room: don't sell SR on UCMP — you'd lose the argument to anyone who knows DMZ Link-Bandwidth. Sell it on the steering, protection and control-plane consolidation that destination-based IP forwarding structurally cannot provide. And we get all three while staying on the OSPF we already run — no IS-IS forklift required.

3 - SR-MPLS vs SRv6: the data-plane choice for this net

Same SR control plane (OSPFv2-SR, prefix-SID 900001..., adj-SID). Different forwarding header. The whole decision is encapsulation overhead, silicon, and how a packet survives an inter-AS hop.

SR-MPLS pushes a 4-byte label per segment. SRv6 carries a 40-byte IPv6 outer + 8-byte SRH base + 16 bytes per SID. The control plane (OSPFv2-SR, the SID plan) is the same either way - only the bytes on the wire differ. Honest footnote for the room: the active/first SID also rides in the IPv6 Destination Address (RFC 8754), so some stacks omit SID[0] from the SRH and carry 40+8+16x(N-1) = 80 B at N=3; the 40+8+16N = 96 B figure used here is the conservative, vendor-quoted upper bound - either way SR-MPLS's 12 B wins decisively.

3a - The byte math, generalised: where SRv6 hurts and uSID claws it back

Overhead is a function of segment count N. SR-MPLS grows 4N; SRv6 classic grows 40 + 8 + 16N; SRv6 uSID packs up to 6 micro-SIDs into one 128-bit carrier so the SRH itself can disappear for short paths.

  N (segments)     1      2      3      4      5      6
  SR-MPLS  = 4N    4      8     12     16     20     24      bytes
  SRv6 classic     64     80     96    112    128    144     bytes   (40 IPv6 outer + 8 SRH base + 16N SIDs)
  SRv6 uSID*       64     64     64     64     64     64      bytes   (40 IPv6 + 8 SRH + 1x16B carrier holding up to 6 uSIDs)
  * uSID carrier = 32-bit block + up to 6 x 16-bit uSIDs in one 128-bit container (RFC 9800, NEXT-C-SID flavor).
    With <=6 uSIDs the SRH can be omitted (uSIDs ride in the IPv6 DA), dropping a further 8 B -> ~56 B floor.
    At N=1 classic already equals the uSID floor (40+8+16 = 64 B); uSID's win is only on multi-segment paths.

uSID is real and closes most of the gap for short paths - but its floor is still the 40-byte IPv6 outer that SR-MPLS never pays. For a latency/MTU-sensitive trading core, "even uSID's best case is ~64 B vs our 12 B" is the line that ends the debate.

3c - The trading-net argument: MTU headroom and serialisation latency

Two numbers a trading desk cares about: how much of a WAN MTU the SR header eats, and the serialisation delay that header adds at line rate. Both favour the 12-byte stack.

  serialisation = header_bits / line_rate
  SR-MPLS  12 B = 96 bits :  96 / 400e9  = 0.24 ns/hop   |   96 / 100e9  = 0.96 ns/hop
  SRv6     96 B = 768 bits:  768 / 400e9 = 1.92 ns/hop   |   768 / 100e9 = 7.68 ns/hop
  Over a 6-hop A/B market-data path on the 100G P2-plane: 6 x 6.72 ns = ~40 ns of pure header serialisation
  that SRv6 adds and SR-MPLS does not. Tick-to-trade budgets are measured in single-digit microseconds - this is
  NOT the dominant term (fibre + queueing + switch pipeline dominate), but on a deterministic feed it is free
  latency we choose not to spend.

The honest scale: header serialisation is nanoseconds, not the headline latency term (that's fibre + queueing + switch pipeline). The decisive point is not the ns - it is MTU determinism: on a 1500 B WAN segment SRv6's 96 B is a real fragmentation/PMTUD risk on an A/B feed, and we gain nothing by taking it.

3d - Inter-AS: globally-routable SIDs vs label stitching (Strawman B)

SRv6's headline inter-domain advantage is real - a SID is an IPv6 address, routable across the NYC/CORE/LON ASBRs with no per-domain label state. Set against it: our actual design uses binding-SID stitch on SR-MPLS, which is mature on EOS today; SRv6 inter-AS on EOS R-series is not (to verify).

  Inter-AS label-stitch walk (SR-MPLS, Strawman B), packet NYC -> LON:
    at PE-NY     : push  <BSID-to-CORE> , <SID-list within NYC to ASBR1>          (labels from NYC SRGB)
    at ASBR1     : pop NYC labels; resolve BSID -> SR policy steers into CORE SRGB SID-list toward ASBR2
    at ASBR2     : resolve next BSID -> LON SRGB SID-list, bottom label = PE-LON prefix-SID
    at PE-LON    : pop, deliver
  Three SRGBs (e.g. all 900000-965535 but independent label state per domain); the binding-SID is the only
  cross-domain contract - it indexes a policy, hiding the far domain's internal SID-list behind one label.
  This is BGP SR Policy (BGP SR Policy SAFI, draft-ietf-idr-sr-policy) + BGP-LS topology feed, controller-driven
  (EOS has no PCEP). Available on EOS today.

Verdict for this net: SR-MPLS. The inter-AS "globally routable SID" win is genuine but it is the one place SRv6 would have helped most, and it is exactly where EOS R-series SRv6 maturity is least proven (to verify) - while binding-SID stitch over BGP SR Policy is available now. We keep the win we can ship.

3A - SRGB and the SID→label arithmetic (deterministic, not vendor magic)

SR-MPLS forwarding labels are derived, not allocated by signalling. The math is identical on every vendor - this is why "no MPLS control plane (no LDP/RSVP)" is a true statement, not a slogan.

   segment-routing mpls

   ! advertise THIS node's prefix-SID as an INDEX
   ! (label index, not an absolute label)

3A - The two gaps, and why neither blocks the design

Being a senior designer means naming what EOS does NOT do for us and showing the workaround, not hiding it. There are exactly two, both already foreshadowed in the matrix.

Gap 1 - Flex-Algo is IS-IS-only on EOS

EOS Flexible Algorithm (4.31.2F, RFC 9350) runs under IS-IS. There is no OSPF Flex-Algo on EOS. We stay on OSPF, so per-class topologies (e.g. "low-latency-only plane") cannot be expressed as a Flex-Algo.

Per-class steering moves from the IGP into a controller-driven SR-TE colour policy (BGP SR Policy, 4.22.1F). The colour-to-plane mapping and computed SID-lists are built out in the steering section - here the point is only that the gap has a clean home.

Gap 2 - No PCEP on EOS

EOS does not speak PCEP, so there is no stateful PCE pushing LSPs the classic way. SR-TE policies are instead programmed via BGP SR Policy SAFI (draft-ietf-idr-sr-policy / RFC 9256), with the controller fed by BGP-LS (4.24.2F).

PCEP would be one transport for the same intent. BGP SR Policy is the IETF-blessed alternative and is what EOS implements - the headend still gets an explicit SID-list, the controller still does the computation. We flag: whether CloudVision or a 3rd-party controller performs the inter-domain path computation for Strawman B is an open item to confirm with Arista.

Feasibility verdict: OSPF-SR + TI-LFA are GA on EOS 4.33.x R-series - no OSPF→IS-IS forklift, no platform swap. The two gaps are addressed by SR-TE colour policies (for steering) and BGP SR Policy/BGP-LS (in place of PCEP), both shipping EOS. The design uses what they already own.

Turning on SR-MPLS (OSPF-SR) — what the IGP advertises

No forklift. OSPFv2 stays; SR turns on as three Opaque LSAs that ride the existing flooding. RFC 7684 = the Extended-Prefix/Link opaque containers; RFC 7770 = the Router-Information opaque LSA; RFC 8665 = the SR sub-TLVs carried inside them.

Why three LSAs, not one: capabilities (RI) flood domain-wide; prefix bindings (Ext-Prefix) follow the prefix into other areas via ABR re-origination; link bindings (Ext-Link) are area-local and never need to leave the area. Opaque type numbers are RFC 7684 / RFC 7770 ground truth; the SR sub-TLVs are RFC 8665. Note: the OSPFv2 Prefix-SID has no N (node) flag — that is an IS-IS construct (RFC 8667); in OSPF a loopback host prefix is simply advertised with NP set so the penultimate hop does PHP.

SRGB scope, ABR re-origination, MSD, and the EOS turn-on

Three operational realities behind the byte layout, then the config that lights it up on Arista R-series. Strawman A is a single area 0, so re-origination is shown for the multi-area / Strawman-B case to keep it pivot-ready.

ABR re-originates the Prefix-SID across areas

The ABR generates the Type-3 summary and re-originates the Extended-Prefix Opaque LSA into Area 1 (area-local scope, route-type 3) carrying the same index 1 (RFC 8665 inter-area). The index is global, so the end-to-end node-SID label is unchanged across the area boundary. Lose this step and the far area cannot label-switch to PE1.

SRGB must be domain-wide; MSD bounds the stack

  • SRGB 900000-965535 on every node. A global index only resolves to one label if base+range match everywhere. Mismatched SRGBs mean each node derives a different label for the same prefix -> broken LSPs. EOS default SRGB is base 900000, size 65536 fleet-wide (verified Arista), so keep the default.

4 - TI-LFA on the core: the failure we are protecting

Strawman A dual-plane core, single OSPF area 0, Arista R-series, OSPF-SR on. Prefix-SID plan: PE1 idx1 = label 900001, PE2 idx2 = 900002, P1 idx3 = 900003, P2 idx4 = 900004 (SRGB base 900000, default EOS SR range 900000-965535). We protect PE1's primary link to P1 - the low-latency leg carrying live market-data / order flow. The question TI-LFA answers locally, with no signalling: when that link drops, what label stack does PE1 push so the very next packet rides the post-convergence path with zero micro-loop?

4 - Step 1: what path does the IGP install AFTER it reconverges?

TI-LFA does NOT run plain LFA. It removes the protected resource from the
topology, then runs SPF on that pruned graph (the "post-convergence" SPF) so the
repair lands on exactly the path the IGP will install after it reconverges.

  Remove link PE1-P1.  Re-run SPF rooted at PE1 toward PE2:
    candidate A:  PE1 -P2-> PE2            = 30 + 30      = 60
    candidate B:  PE1 -P2-> P1 -> PE2      = 30 + 50 + 10 = 90
    post-conv path = min(A,B) = A = 60  ->  PE1 -> P2 -> PE2  (rides P2 plane)

Why post-convergence matters: a naive LFA could pick any loop-free neighbour, but if that differs from where the IGP settles, traffic moves twice (LFA path, then re-route after SPF) - a transient micro-loop and a second hit. TI-LFA's repair IS the post-convergence path, so the packet is placed on its final path immediately. Each plane protects the other: P1-plane failure spills onto the P2 (capacity) plane, which is exactly the engineered fallback.

4 - Step 2: P-space, Q-space, and is there a PQ-node?

P-space(PE1, \ PE1-P1) - nodes PE1 reaches WITHOUT the failed link on their shortest path:
   P2 : PE1's shortest path to P2 = PE1-P2 (30). Does not cross PE1-P1.  -> IN
   P1 : PE1's shortest path to P1 (full topo) = direct PE1-P1 (10). Crosses failed link. -> OUT
   PE2: PE1's shortest path to PE2 (full topo) = PE1-P1-PE2 (20). Crosses failed link.  -> OUT
   => P-space = { P2 }

Q-space(PE2, \ PE1-P1) - nodes that reach PE2 WITHOUT the failed link (reverse SPF rooted at PE2):
   P2 : P2-PE2 (30), clean.  -> IN
   P1 : P1-PE2 (10), clean.  -> IN     (the failed link is PE1-P1, not P1-PE2)
   => Q-space = { P1, P2 }

   P-space ∩ Q-space = { P2 }  AND  P2 is on the post-conv path  ->  P2 is a PQ-node.

Repair-list rule (TI-LFA, draft-ietf-rtgwg-segment-routing-ti-lfa §6.2; P-/Q-space defined in RFC 7490 RLFA): a remote node in both P-space and Q-space and on the post-convergence path -> the repair list is a single node segment to that PQ-node. The dual-plane core is built so a PQ-node always exists - the opposite plane's P-node sees both sides of the break. That is the design payoff: one label repairs the failure, versus the square / ladder case (next slide) where P-space and Q-space do not intersect and a 2-SID stack is needed.

4 - Step 3: the repair SID list, with real labels

Repair list at PE1 (PLR) for destination PE2, link-protect on PE1-P1:
   active SID stack = [ 900004 , 900002 ]   (repair SID 900004 -> P2 PQ-node; then 900002 -> PE2)
   backup next-hop  = P2  via interface to the P2 plane (PE1-P2)

Imposed stack uses PE2's Node-SID 900002 (global, prefix-SID for PE2's loopback) under the
repair SID 900004. With a PQ-node, TI-LFA prepends ONE repair label (900004) onto the normal
SR label for the destination.

Label walk, hop by hop (PE2's prefix-SID advertised NP=0 => PHP at the penultimate hop):
   PE1 : push 900004 over 900002       -> out PE1-P2     [stack: 900004 | 900002 | payload]
   P2  : 900004 = P2's own Node-SID    -> POP outer (PHP for P2), continue SR to PE2 [900002 | payload]
   P1* : (P2 is penultimate to PE2)    -> POP 900002 (PHP for PE2) before delivery   [payload]
   PE2 : receive IP/labelled payload, deliver to local prefix

Result: packet rides PE1 -> P2 -> PE2 (post-conv path, cost 60) on the FIRST packet after
failure detection. Two labels of MSD consumed at imposition (repair SID + destination Node-SID).

Because the PQ-node P2 sits on the post-convergence path, the repair adds a single extra label (P2's Node-SID 900004) on top of the destination's own Node-SID 900002 - P2's SR forwarding then carries the rest. Contrast the square/ladder topology where P-space and Q-space do not intersect: TI-LFA falls to [Node-SID(P), Adj-SID(P→Q)] when P and Q are adjacent (§6.3), or a list of Adj-SIDs when P and Q are distant (§6.4) - more labels, more MSD. The dual-plane design deliberately yields the minimal single-repair-SID case.

4 - The TLVs the repair is computed from (OSPFv2-SR LSDB)

Everything TI-LFA needs is already in the OSPFv2-SR LSDB - no T-LDP, no PCEP, no extra
sessions. RFC 7684 defines the Extended Prefix / Extended Link Opaque LSA containers;
RFC 8665 adds the SR Sub-TLVs carried inside them.

P2 originates an OSPFv2 Extended Prefix Opaque LSA (RFC 7684 Opaque type 7) for its loopback /32:
  Extended Prefix TLV
    Route Type   = 1 (Intra-Area)
    Prefix Length= 32        AF = 0 (IPv4 unicast)
    Prefix       = <P2 loopback>/32
    Prefix-SID Sub-TLV (RFC 8665)
      Flags : NP=0 (PHP allowed: penultimate hop pops before delivery to P2),
              M=0  (not from an SR Mapping Server, RFC 8661),
              E=0  (no explicit-NULL),
              V=0, L=0  => SID is an INDEX into the SRGB, not an absolute label
      Algorithm = 0 (SPF)        MT-ID = 0
      SID/Index = 4         ->  label = SRGB_base(900000) + index(4) = 900004

PE1 + every node also originate an OSPFv2 Extended Link Opaque LSA (RFC 7684 Opaque type 8):
  Extended Link TLV  (per adjacency)
    Adj-SID Sub-TLV (RFC 8665)
      Flags : B=1 (Backup - adjacency eligible as a TI-LFA repair segment),
              V=1, L=1 (value is a LOCAL label), G=0 (not a group/parallel adj), P=0
      Weight = 0        SID/Label = <dynamically allocated local label>

Router Information Opaque LSA carries the SR-Algorithm TLV + SID/Label Range (SRGB) TLV.

PE1 computes the repair entirely from these LSAs: P2's Prefix-SID Sub-TLV gives the Node-SID index 4 -> label 900004 (deterministic from the advertised SRGB base 900000); the B-flagged Adj-SID Sub-TLVs mark which adjacencies TI-LFA may use as repair segments in the square / distant cases. With NP=0 the penultimate hop performs PHP; if instead the prefix advertised NP=1 (with E=1 for explicit-NULL), the SID would be delivered to P2 intact - a flag choice, not a fabrication. No state is signalled into transit nodes - they do ordinary SR label forwarding. This is the structural win over RSVP-TE FRR (per-LSP bypass state) and RLFA (T-LDP sessions to the PQ-node).

Inter-AS stitch (1/3): the problem and the per-domain SID plan

Strawman B: R-NY (NYC AS65001) must reach PE-LN (LON AS65002) across three OSPF domains joined by eBGP at ASBRs. No single IGP, no PCEP on EOS. A binding-SID-stitched SR Policy hides each domain's internals behind one label.

Inter-AS stitch (2/3): the head-end's 3-label stack and the hop-by-hop walk

  LN1   binds  BSID 902801  -> policy "across LON, LN1 to PE-LN"  seg-list = [ 902031 ]          (PE-LN node-SID; LN1 is the policy owner)

                       node-SID   BSID-core  BSID-lon

The mechanism is RFC 9256 §8.3 (incoming active SID is a BSID): at the node owning the BSID's policy, a packet whose top label is BSID B has B popped and the policy's segment list pushed, then is forwarded per the first segment. So at CORE, CR1 swaps BSID 901801 → [901019]; at LON, LN1 swaps BSID 902801 → [902031] → PE-LN. The BSID is the only thing the upstream domain sees; the intra-domain SID list is private. 6.1 - The UCMP problem: ECMP hashes by flow, not by capacity

Strawman A. PE1 (market-data edge) -> PE2 (trading edge), two disjoint planes, but the planes are NOT equal capacity.

Both planes cost OSPF metric 10 end-to-end (PE1->Px = 10, Px->PE2 = 10, so each path = 20), so OSPF installs them as ECMP. EOS hashes per 5-tuple flow into equal shares: ~50/50. The 100G plane is offered 125G of a 250G demand and tail-drops; the 400G plane sits at 125/400 = 31% full. Same total path cost, wrong distribution. (Aggregate-demand model; the real fix is to make the share track capacity, not metric.)

6.2 - Why it splits 50/50: the ECMP share model in the FIB

ECMP is capacity-blind by construction. The FIB hashes a flow's 5-tuple into a fixed share array and gives each next-hop an equal number of shares.

Key: a flow is pinned to one share, so per-flow ordering is preserved (vital for market-data) but the split ratio is set by share COUNT per next-hop, not by link speed. To fix UCMP we must change the share counts, not the metric. (Aggregate-demand model: real per-flow loads vary with the hash and flow-size distribution; the equal-share count is what we are correcting.)

6.3 - Target ratio: weight proportional to capacity

We want each plane loaded to the same utilisation. Solve for the split that makes both planes equally full.

Capacity-proportional weight (the target):

  C(P1) = 400G   C(P2) = 100G   total = 500G
  w(P1) = 400 / 500 = 0.80
  w(P2) = 100 / 500 = 0.20            -> 80 / 20

Apply to the 250G demand:
  P1 = 250 x 0.80 = 200G  on 400G = 50.0% util
  P2 = 250 x 0.20 =  50G  on 100G = 50.0% util
  => identical utilisation, no congestion.

Headroom check (single demand spikes to 320G):
  P1 = 320 x 0.80 = 256G  (400G) = 64%
  P2 = 320 x 0.20 =  64G  (100G) = 64%   still safe.

Break point (when does P2 hit 100G?):
  0.20 x D = 100  ->  D = 500G  = full fabric.
  So 80/20 keeps both planes feasible up to line-rate
  aggregate. ECMP's break point was D = 200G (100/0.5).

Capacity-proportional weight = C(link)/sum(C). It equalises utilisation but ignores latency; here the low-latency plane is the 400G one, so 80/20 happens to push the bulk onto the fast plane too. (If the latency-critical plane were the small one, you would steer the latency-sensitive class explicitly with SR-TE rather than bulk-balance it.) Two ways to install the weight: SR-TE policy weights inside a domain you own (6.4); BGP link-bandwidth across a boundary you don't (6.5).

6.4b - How the FIB realises 80/20: integer repetition counts

Hardware cannot store a fractional weight. EOS BGP UCMP "mode 1" resolves weights to an INTEGER repetition count per next-hop across an N-entry array (VERIFIED: Arista BGP UCMP, EOS 4.24.1F+). An SR-TE policy's w/Sw weights are realised the same way - as integer next-hop shares in the FIB. The arithmetic is exact and worth showing.

Weights 80:20 -> reduce -> 4:1
Sum of repetitions N = 4 + 1 = 5  (must be <= the max
ECMP/UCMP array width the silicon supports).

FIB programs next-hop array of 5 entries:
   [ P1, P1, P1, P1, P2 ]
   share(P1) = 4/5 = 0.80
   share(P2) = 1/5 = 0.20            EXACT, no rounding error.

If weights were 70:30 -> 7:3, N = 10:
   [ P1 x7 , P2 x3 ] -> 0.70 / 0.30  exact.

If weights were 65:35 -> 13:7, N = 20  (still small).
Arbitrary ratios round to nearest 1/N; pick N >> ratio
granularity. With a 256-wide array:
   80/20 -> 205 : 51  (205/256=0.801, ~0.1% error).

Per-flow ordering is preserved (each 5-tuple still pins to one next-hop), so UDP market-data is not reordered within a flow. STILL OPEN (do not promise a granularity): exact max-SID-depth (MSD) and max ECMP/UCMP array width on the R-series silicon - the node advertises its imposition MSD per RFC 8476 (OSPF MSD sub-TLV) so a controller will not push a SID-list deeper than the platform can impose; lab to confirm the actual R-series MSD and array width before committing a fine-grained ratio.

6.5 - Fix across a boundary you do NOT own: BGP Link-Bandwidth

In Strawman B the next-hop planes sit behind an ASBR in a neighbour AS (e.g. CORE/WAN AS65000 reaching LON AS65002). You cannot write an SR Policy into someone else's domain. Instead, attach the capacity as a BGP signal: the Link-Bandwidth extended community. The receiver turns it into UCMP next-hop weights.

Link-Bandwidth extended community (draft-ietf-idr-link-bandwidth):
  Type    0x40 = transitive  /  0x00 = non-transitive   (high octet)
  SubType 0x04 = Link-Bandwidth   -> transitive type 0x4004, non-transitive 0x0004
  Global Admin   2 octets = local AS number (or any 2-octet value)
  Local Admin    4 octets = bandwidth as IEEE-754 float32, BYTES per second (NOT bits)

  Plane via ASBR-a (400G): 400e9 bit/s = 50.0e9 byte/s  -> float32 = 0x513A43B7  (decodes 49,999,998,976 ~= 50e9)
  Plane via ASBR-b (100G): 100e9 bit/s = 12.5e9 byte/s  -> float32 = 0x503A43B7  (decodes 12,499,999,744 ~= 12.5e9)
  (hex VERIFIED by IEEE-754 encode; float32 cannot hold 50e9 exactly - the ~24-bit mantissa rounds, which is fine: ratio is what matters)

router bgp 65000
   maximum-paths 64 ecmp 64
   ucmp mode 1                         ! VERIFIED feature; FIB = integer repetition counts (see 6.4b)

Mechanism: RX receives both paths, sums the bandwidths (Sw = 50e9 + 12.5e9 = 62.5e9 B/s) and programs next-hops 50/62.5 = 0.80 and 12.5/62.5 = 0.20 -> the same 4:1 array as 6.4b, but signalled, not policy-pushed. DESIGN CAVEAT (senior-level): link-bandwidth is a LOCAL, relative quantity. Sending it TRANSITIVE across many ASNs is semantically weak - an AS three hops away cannot meaningfully act on AS65001's absolute byte/s. In practice use the NON-TRANSITIVE form (0x0004) between the two ASNs that share the parallel links, or re-originate a locally-meaningful value at each ASBR. EOS lets you aggregate received bandwidth from downstream and re-split it upstream, which is the cleaner inter-AS pattern.

7 - Steering one service onto a path: why ECMP can't, and what does

Strawman A. Two services share PE1→PE2 but want different paths. Market-data (colour 100) must take the low-latency P1 plane; bulk takes shortest-path/ECMP.

The forwarding decision is keyed on destination prefix only. Market-data and bulk both head to PE2's loopback, so the IGP/FIB gives them one identical next-hop set. There is no field in a plain IGP route that says "this flow is latency-class." That is the gap SR-TE closes. (With these metrics the P1 plane is strictly lower-cost, so unequal-cost; the ECMP-split case is the equal-cost variant called out on the next slide.)

7 - Why ECMP, plain IGP and (on OSPF) Flex-Algo are the wrong tools

Three candidate mechanisms, and why only SR-TE colour policy expresses per-class path on this network.

MechanismWhat it keys onCan it pin market-data to the P1 plane?
Plain IGP shortest-pathdestination prefix → lowest metric No. One best path per prefix. Both classes follow the same lowest-metric route; you cannot split by class.
ECMP (equal-cost)per-flow hash over equal-cost next-hops No. If P1 and P2 planes were made equal-cost, ECMP would split market-data across both - the opposite of deterministic. It load-balances, it does not classify.
Flex-Algo (per-algo metric/plane)prefix-SID bound to an algorithm (e.g. min-delay) Architecturally yes - but Flex-Algo is IS-IS-only on EOS. We are staying on OSPFv2, so this door is closed (verified Arista fact).

Because EOS has no OSPF Flex-Algo, the only OSPFv2-compatible way to express "this class, that plane" is an explicit-path SR-TE policy selected by colour. SR-TE rides on the same OSPF-SR prefix-SIDs already turned on in section 5 - no new IGP, no forklift.

RFC refs: SR Policy architecture and automated steering = RFC 9256; Colour Extended Community / Tunnel Encapsulation attribute = RFC 9012 (extended for SR-Policy steering by RFC 9830); BGP advertisement of SR Policies (SAFI 73) = RFC 9830 (was draft-ietf-idr-sr-policy).

7 - The colour-100 policy: endpoint, SID-list, and the label math

SR Policy is the tuple <headend=PE1, colour=100, endpoint=PE2 10.0.0.2>. Its single candidate path is an explicit SID-list that pins the P1 plane.

SID-list (prefix-SIDs, node steering) = [900003  →  900002] = "via P1, then to PE2." Both are global prefix-SIDs (node-SIDs), so each is reachable by shortest path from the prior hop; the headend need not enumerate adjacency-SIDs unless a specific parallel link must be pinned. Here PE1 has a single link to P1, so the P1 node-SID alone forces the low-latency plane.

SID -> label (SRGB base 900000, default EOS range 900000-965535; verified)
  index  3  (P1)   ->  900000 + 3  = 900003
  index  4  (P2)   ->  900000 + 4  = 900004   (deliberately NOT in the list)
  index  2  (PE2)  ->  900000 + 2  = 900002

Colour-100 SID-list  = [ 900003 , 900002 ]      (push order: top = 900003)
SID-list depth (MSD)  = 2 labels  ->  see MSD check on next slide

Index-to-label is deterministic: prefix-SID index + SRGB base. 900004 (P2) is intentionally absent - that is how the policy excludes the high-latency plane. This is path expression by SID composition, not by metric tuning.

7 - The steering trigger: colour extended community on the service route

The policy exists - now market-data traffic must select it. Automated steering (RFC 9256): a BGP route carrying a colour extended community matching the policy's <colour, endpoint> resolves its next-hop THROUGH the policy.

! On the originating PE / route-reflector: tag the market-data service route with colour 100
ip prefix-list MKTDATA-PFX seq 10 permit 10.0.0.2/32
route-map SET-MKTDATA-COLOUR permit 10
   match ip address prefix-list MKTDATA-PFX
   set extcommunity color 100 additive          ! attach Color:100  (RFC 9012; EOS 4.23.1F)
route-map SET-MKTDATA-COLOUR permit 20          ! bulk + everything else: no colour
! applied to the market-data service prefixes only; bulk prefixes get no colour.
! NB: for a coloured route to steer, its BGP next-hop must already be IGP-resolvable (Arista doc).

Colour-ext-comm in route-map set clauses is supported since EOS 4.23.1F (verified TOI): set extcommunity color <n> additive. The matching/steering semantics (colour + endpoint = next-hop) are RFC 9256/9012 and are deterministic.

8 - Multicast: why it sits OFF the SR fabric (back-pocket)

SR-MPLS is unicast source-routing. It steers ONE packet down a label stack to ONE egress. Market-data is one-to-many. Different forwarding planes - do not force them together. (Raise this only on a segue; it is not part of the core UCMP/steering pitch.)

Design position: the trading market-data feed STAYS in its own PIM-SSM plane. We do not tunnel it through SR-TE. The SR fabric solves unicast UCMP/steering; the SSM plane solves deterministic A/B replication. Two planes, two problems, both solved well.

8.1 - The default: PIM-SSM A/B feeds, kept off SR (and why that's correct)

Market-data is delivered as SSM (S,G) joins driven by IGMPv3. Exchange publishes the same data twice on two disjoint groups - feed A and feed B - and the trading app arbitrates. The replication plane is built bottom-up from receiver joins; it is not a controller path.

SSM has NO rendezvous point and NO shared tree (so no RP failover risk, no (*,G) state, no MSDP). Each (S,G) is a shortest-path tree built straight from the last-hop router to the source. That determinism is exactly why trading desks use SSM - and exactly why it does not want a controller-computed SR path in front of it.

8.3 - The recommendation, in one line

When this comes up in the room, this is the position to hold - and the reason it is the right call for a trading network specifically.

Bottom line: don't tunnel a deterministic A/B market-data feed through a controller-computed unicast fabric just to win an architecture-diagram aesthetic. The replication plane is a solved, separate problem; SR-MPLS earns its keep on the unicast side. If convergence is ever mandated, SR P2MP / Tree-SID is the EOS-supported path (and does not need PCEP - install via CVX/controller or CLI); BIER is architecturally cleaner for a stateless core but is not in the EOS multicast docs, so it is a hard to-verify before anyone commits.
